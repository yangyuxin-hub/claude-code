# 项目演化记录

更新时间：2026-04-01

> 用于记录这个项目是如何一步步搭建、修正、完善的。
> 重点不是只记“做了什么”，而是记录：
> - 当时怎么想
> - 做了什么判断
> - 为什么改变方向
> - 背后的考虑是什么
> - 后来哪些认识被验证，哪些需要继续修正

---

## 记录原则

1. **记录转变，不只记录结果**
   - 不只写“最后变成了什么”
   - 更要写“原来怎么想、后来为什么改”

2. **记录判断背后的原因**
   - 是因为数据反馈？
   - 是因为实际执行卡住？
   - 是因为更看清了用户、市场、变现路径？

3. **记录系统如何被使用逼着进化**
   - 好的系统不是一开始设计完美
   - 而是在真实使用中，不断暴露问题、不断修正、不断迭代出来的

4. **记录可复用的方法论**
   - 哪些调整未来还会再用
   - 哪些经验值得沉淀成 SOP、判断框架或对外表达

## 2026-04-01｜第一次系统性对齐

### 1. 项目定位从“公众号运营”转向“Claude Code 个人 IP / 公众号体系建设”

**原来的模糊状态：**
这个工作区一开始更像“公众号助手”或泛内容运营助手，虽然已经有 Claude Code 课程相关内容，但身份和目标没有完全收拢。

**新的判断：**
项目的核心不是泛公众号运营，而是围绕 Claude Code 做内容增长、案例沉淀、信任建立和后续转化。

**为什么会有这个转变：**
因为实际项目目录、文章结构、课程文档、案例方向都已经明显表明，真正主线是 Claude Code，而不是泛内容运营。

**背后的考虑：**
如果身份不收拢，后面做选题、内容、案例、变现判断时都会发散；只有先把主线拉直，后续动作才会统一。

---

### 2. 阶段目标明确为“涨粉优先”

**原来的潜在方向：**
项目里同时有课程、案例、公众号、社区、变现等多条线，容易平均用力。

**新的判断：**
现阶段第一目标不是马上变现，而是先涨粉、先打认知、先让更多人认识这个账号和这套表达。

**为什么会有这个转变：**
因为在冷启动阶段，没有足够关注和信任，课程、社区、咨询都很难有效承接。

**背后的考虑：**
先用内容建立传播性和关注度，再谈后端转化，效率更高。

---

### 3. 平台分工明确：小红书拉新，公众号沉淀

**原来的状态：**
已经意识到需要双平台，但分工还没有彻底写成默认规则。

**新的判断：**
- 小红书负责拉新、破圈、测试传播性
- 公众号负责沉淀深度内容、建立信任、承接后续转化

**为什么会有这个转变：**
因为两个平台的内容逻辑和用户预期不同，不能简单一稿两发。

**背后的考虑：**
同一个主题可以双版本分发，但每个平台要服务不同目标。

---

### 4. 内容判断标准从“体系完整”转向“传播优先”

**原来的风险：**
容易按课程结构、知识体系顺序去推进内容，导致内容虽然完整，但传播力不一定强。

**新的判断：**
在当前阶段，优先级应是：
**传播性 > 体系完整性**

更具体地说：
- 热点/事件型优先
- 强观点认知型优先
- 真实案例型优先
- 教程和系统讲解可以往后排

**背后的考虑：**
冷启动阶段最宝贵的是注意力，不是内容目录完整。

---

### 5. Skill 使用策略被重新收拢

**原来的状态：**
可能存在多套工具、多个旧方法并行，浏览器操作也有稳定性问题。

**新的判断：**
- 有 Claude Code skill 时，优先参考 `C:\Users\18113\.claude\skills`
- 浏览器/网页访问默认优先使用 `web-access`
- 当前核心 skill 组合聚焦在：
  - `topic-eval`
  - `web-access`
  - `baoyu-xhs-images`
  - `baoyu-markdown-to-html`
  - `baoyu-post-to-wechat`

**为什么会有这个转变：**
因为工具太散会让流程变乱，稳定主链路比堆技能更重要。

**背后的考虑：**
当前最重要的是形成稳定工作流，而不是拥有尽可能多的能力。

---

### 6. 飞书多维表格的角色被明确

**确认结果：**
Base 已打通，可访问。

**新的结构理解：**
- `数据表`：记录已经发布和准备发布的内容
- `素材库`：记录待消化素材、灵感来源、候选选题

**背后的考虑：**
内容生产如果没有稳定的素材池和发布池，后面越做越乱；先明确数据结构，后面就能在飞书里真正跑流程。

---

### 7. 商业路径被重新校正：不只盯课程和社区

**原来的默认想象：**
项目名称和部分文档容易让人自动联想到“公众号 → 课程 → 社区”。

**新的判断：**
课程和社区可以做，但不是唯一方向，也不是当前必须优先。

更现实的中短期路径可能是：
- 高客单定制服务
- 1对1陪跑
- 咨询服务
- 课程（可选）
- 社区（可选）

**为什么会有这个转变：**
因为用户明确表达了更偏好高价值、可复制、能落地的商业路径，而不是被“卖课/做社区”锁死。

**背后的考虑：**
对当前阶段来说，内容体系建设的意义不仅是涨粉，也是为高客单转化做信任铺垫。

---

### 8. 行业偏好被明确：金融、电商优先

**新的判断：**
优先关注金融、电商领域的 AI 赋能。

**原因：**
- 付费能力更强
- 流程更清晰
- 场景更容易标准化
- 产品和服务更容易复制

**背后的考虑：**
不是所有行业都值得平均用力。优先在更容易成交、也更容易标准化的行业建立样板，更现实。

---

### 9. 项目过程本身被确认为“值得记录的资产”

**新的判断：**
不只是最终文章、案例、课程值得沉淀，整个“公众号创作体系是怎么一步步搭起来的”也值得记录。

**为什么重要：**
因为一个好系统不是凭空设计出来的，而是在使用中不断发现问题、修正判断、优化流程、逐渐打磨出来的。

**未来价值：**
这部分过程记录，将来可以变成：
- 方法论
- 复盘内容
- 对外表达素材
- 服务客户时的可信经验

## 后续记录建议

后面每次有下面这些变化，都建议继续补进来：

1. **目标变化**
   - 比如：从涨粉优先变成成交优先
   - 或从泛认知转向行业案例优先

2. **内容策略变化**
   - 哪种标题更有效
   - 哪类题材更容易传播
   - 哪个平台反馈更好

3. **产品/服务方向变化**
   - 为什么决定加强高客单服务
   - 为什么决定暂缓课程或社区

4. **系统迭代变化**
   - 素材怎么管
   - 选题怎么排
   - 发布流程怎么优化
   - 哪些环节被自动化替代

5. **认知变化**
   - 原来以为对的，后来发现不对
   - 原来没重视的，后来发现特别关键
